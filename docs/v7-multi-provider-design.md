# dsgn Multi-Backend Design Doc — non-Claude model backends

Verified against `src/main/agent.ts` (the `Session` interface, `startSession`/`query()` loop, `canUseTool` gate, `AUTO_ALLOW_TOOLS`/`EDIT_TOOLS`/sidecar guard) and `src/shared/api.ts` (`AgentEvent`, `AgentOptions`, `PermissionRequest`, `PermissionMode`). The seam is real and narrow exactly as described: everything provider-specific lives inside `startSession`; the IPC layer, preload bridge, and all four zustand stores are backend-agnostic.

**The clean split:** OpenAI / Gemini / Grok are interchangeable behind a single AI-SDK `streamText` loop. v0 is a category mismatch — it does not satisfy the agentic-edit contract and must not be a `ModelProvider`.

## 2. The provider seam

Introduce a `ModelProvider` factory that produces a `ProviderSession` emitting the **same `AgentEvent` union** and honoring the **same control methods** as today. The existing `Session` interface (`agent.ts:70-85`) is already 90% of this — formalize it and add the optional live-control methods that the IPC handlers currently call directly on `session.query`.

```ts
// src/main/backends/types.ts
interface ProviderSession {
  key: string                                   // projectKey(root)
  root: string
  options: AgentOptions                         // { model?, effort?, permissionMode? } + provider?
  input: { push(text: string): void; close(): void }  // streamed multi-turn input
  abort: AbortController
  pending: Map<string, PendingPrompt>           // { toolName, settle('allow'|'deny') }
  emit: (e: AgentEvent) => void                 // auto-tags projectKey; no-op once disposed
  dispose(): void                               // stop emitting only (the `disposed` guard)
  // optional live controls — IPC already optional-chains these:
  setModel?(model: string): Promise<void>
  setPermissionMode?(mode: PermissionMode): Promise<void>
  interrupt?(): Promise<void>
}

interface ModelProvider {
  id: string                                    // 'claude' | 'aisdk'
  startSession(root: string, options: AgentOptions,
               getWindow: () => BrowserWindow | null): Promise<ProviderSession>
}
```

**Dispatch.** Add `provider?: string` to `AgentOptions` (or route off a model prefix: `gpt-*`/`gemini-*`/`grok-*` → `aisdk`, else `claude`). The current `startSession` body becomes `claudeProvider.startSession`. A new `aisdkProvider.startSession` builds the `streamText` loop. `agent:open-project` picks the provider; the `sessions` Map / `activeKey` / LRU / `projectKey` tagging / every IPC handler / preload / store stay byte-for-byte unchanged.

**Must-emit:** `delta`, `status`, `done` (exactly one per turn — interrupt and clean completion both emit `done`), `error`. Never emit a "start" — the renderer calls `startAssistant()` itself on send (`ChatPanel.tsx`).
**Should-emit if supported:** `permission-request`+`permission-resolved` (any mutating/shell action), `commands` (the `/` menu — Claude-only source).
**Must honor:** `input.push` as a turn-enqueue on a persistent session; the `disposed` guard so replaced sessions go silent; sidecar (`.dsgn/`) hard-deny; teardown semantics in `closeSession`.

The AI-SDK→AgentEvent mapping is mechanical: `fullStream` `text-delta`→`delta`, `tool-call`→`status` (`describeTool` reused), `tool-approval-request`→`permission-request`, `finish`→`done`, `error`→`error`. `streamText({ abortSignal })` covers `interrupt`; permission modes collapse into the `toolApproval` resolver closure reading a mutable `mode`.

## 3. Trade-offs — what a non-Claude AI-SDK backend loses

**Lost (Claude-Agent-SDK-only, no portable equivalent):**
- **Skills (`.claude/skills`) auto-discovery** via `settingSources` — gone. No equivalent in any provider.
- **Repo `CLAUDE.md` auto-apply** — only best-effort recoverable by reading the file (walking up the tree) and concatenating into a hand-authored `system` prompt.
- **Slash-commands menu** (`commands` event, sourced from SDK `system/init`) — no source; menu stays empty or hardcoded.
- **The built-in, hardened tool suite** (Read/Edit/Write/MultiEdit/Bash/Glob/Grep/LS) and their prompt-tuned descriptions, plus the `claude_code` system preset.

**Must be rebuilt (the long pole, ~2–3 days, security-sensitive):**
- **The toolset** as AI-SDK `tool()` defs with Zod schemas: path-sandboxing (traversal defense), the string-replace edit semantics, ripgrep/fast-glob wrappers, Bash exec with cwd/timeout/output-truncation/command-injection guards. The Claude SDK hardened all of this for free.
- **Streaming-input → per-turn-messages translation:** AI SDK is request/response per turn; the push-driven `InputStream` collapses into a `ModelMessage[]` the session mutates and replays each turn.
- **The permission loop:** `canUseTool` → wrap each tool's pre-execution in the `toolApproval` resolver: sidecar hard-deny, `AUTO_ALLOW_TOOLS` auto-allow (+status line), `EDIT_TOOLS` honored under `acceptEdits`, else emit `permission-request` and await. The card-release-on-permissive-switch behavior must be replicated for parity.
- **A dsgn system prompt** (file-editing discipline, tool-selection guidance, repo conventions) — no `claude_code` preset.

Net: ~6–8 engineering days for a working multi-provider edit-the-repo chat, with the **accepted, explicit regression** that skills + slash-commands + native CLAUDE.md discovery are Claude-only. select-to-edit (`data-dsgn-source` → prompt anchoring) is provider-agnostic and works on any backend that can edit files.

## 4. Auth reconciliation

dsgn's premise — "each user authenticates with their own Claude subscription" — **does not transfer to any other provider.** OpenAI/Gemini/Grok/v0 are all **bring-your-own-API-key**, billed per token, decoupled from any subscription. This is a real product shift: the backend choice now also implies an **auth mode**.

**Design:**
- Two auth modes coexist. Claude path = OAuth subscription (unchanged). AI-SDK paths = per-provider API key.
- **Key storage:** Electron `safeStorage`/keychain, never committed — consistent with the "auth is per-user at runtime, never commit secrets" convention. A small settings UI captures a key per provider (`OPENAI_API_KEY`, `GOOGLE_GENERATIVE_AI_API_KEY`, `XAI_API_KEY`, `V0_API_KEY`).
- **Model picker carries credential source.** `AgentOptions` gains `provider`; the picker greys out / prompts-to-add-key for providers without a stored key. Selecting a keyless provider raises an "add API key" affordance rather than failing a turn.
- **Set billing expectations explicitly.** API turns cost money per token — not covered by a flat ChatGPT/subscription plan. Surface this once on first non-Claude selection.
- **Auth-error parity:** AI-SDK auth/quota failures must produce `error` messages that either match `isAuthError` (`store.ts:274`) to raise the onboarding banner, or render as plain inline errors. Reuse the existing banner for "missing/invalid key."

## 5. Buildable now vs needs-a-decision

**Build autonomously this session (zero risk to the working Claude path):**

1. **Formalize the `ModelProvider`/`ProviderSession` interface** in `src/main/backends/types.ts`, extracting today's `Session` shape verbatim. No behavior change.
2. **Refactor the current `startSession` into `claudeProvider`** (`src/main/backends/claude.ts`) implementing `ModelProvider`. `agent.ts` keeps the `sessions` Map, `activeKey`, `closeSession`, `resolvePending`, and all IPC handlers — it just calls `provider.startSession`. This is a pure move; `bun run typecheck && bun run verify` must stay green and `agent-e2e.mjs` must still edit the fixture.
3. **Add `provider?: string` to `AgentOptions`** and a `pickProvider(options)` dispatcher that today only ever returns `claudeProvider` (default). Plumb `provider` through `toAgentOptions` so it's a no-op when unset.
4. **Scaffold `aisdkProvider` behind a feature flag** (`DSGN_AISDK=1` / a hidden setting), unreachable from the UI by default: the `streamText` loop skeleton, the `fullStream`→`AgentEvent` mapping, the `toolApproval`→`permission-request` bridge, and a CLAUDE.md-injection helper — wired but with a **stub toolset** (Read + Edit only, sandboxed) so it compiles and a smoke test can drive one delta/done turn without touching the default path.
5. **Key-storage plumbing** via `safeStorage` + a tiny IPC get/set, no UI surfacing yet.

All five are additive and gated; the default runtime is byte-identical to today.

**Defer to the user's explicit decision (do NOT build unprompted):**
- **Committing to the full AI-SDK agent loop** — the hardened toolset (Bash/Glob/Grep/MultiEdit with the full security surface), the production permission-mode resolver, the authored dsgn system prompt, and per-provider real-turn e2e tests. This is the ~6–8 day investment and carries the accepted skills/slash-command regression — a product call, not a refactor.
- **Which providers ship first** (recommend Gemini for lowest friction, then OpenAI for ecosystem uniformity, Grok last given the `.chat` footgun).
- **The v0 `/generate` action** — separate workstream, not a backend; build only if the user wants the 0→1 generation feature.
- **Surfacing the backend picker in the toolbar** and the BYO-key billing UX — flip the flag once the loop is real.

## 6. Risks

- **Regression in the incumbent path during refactor.** The provider extraction touches the most load-bearing file. Mitigation: pure move first, gate everything new, require `bun run verify` + `agent-e2e.mjs` green before and after each step.
- **`disposed`-guard / projectKey leak.** Any new provider that sends to `webContents` directly instead of through `session.emit` will leak events into a reopened project's chat. Contract: all emission routes through `emit`.
- **Security surface of a hand-built toolset.** Path traversal, sidecar bypass via Bash, command injection, unbounded output. The Claude SDK hardened these; the AI-SDK path re-exposes them. Treat the toolset as security-sensitive code with tests, not a quick port.
- **`done`-once invariant.** The setup-flow and `isRunning` both hinge on exactly one `done` per turn at true turn end. AI-SDK multi-step loops can mis-emit; map `finish` carefully and ensure interrupt still emits `done`.
- **Provider footguns:** Grok silently drops client-side tools on the default Responses API (must use `xai.chat`); Gemini rejects `z.union`/`z.record` schemas (audit tool schemas or set `structuredOutputs:false`); GPT-5/o-series require reasoning items to be passed back with tool outputs or the next turn errors.
- **Auth-mode confusion / surprise billing.** Users accustomed to the flat Claude subscription may not expect per-token API charges. Surface cost + key requirement before the first non-Claude turn; map auth/quota errors to the existing onboarding banner.
- **Feature-parity perception.** Non-Claude backends lose skills, slash commands, and native CLAUDE.md — if shipped without clear labeling, users may read it as the app being "worse." Mark Claude as the full-capability path in the picker.

**Decisive recommendation:** Build items 1–5 this session (additive, gated, Claude path untouched). Standardize non-Claude backends on the **Vercel AI SDK** (`streamText`+`stopWhen`) for one uniform loop across OpenAI/Gemini/Grok — do not adopt the OpenAI Agents SDK as a second bespoke path. Treat **v0 as a future `/generate` action, never a backend.** Hold the full AI-SDK agent-loop commitment (the ~6–8 day toolset rebuild + accepted skills regression) for explicit user sign-off.
